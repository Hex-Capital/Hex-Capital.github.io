# Stage -- Naval Ravikant Evaluation

The structural problem with Stage crystallizes in one sentence from the dossier: "The 'structured chapters' concept is a UX innovation, not a deep technical moat." That's not me editorializing — that's the honest assessment of what's being built. A presentation layer on top of GitHub's diff API. When I think about where outlier returns come from, they come from leverage that compounds — code that gets better with every user, networks where each node makes every other node more valuable, knowledge so deep it takes competitors years to acquire. A UX metaphor — however elegant — is none of these things. Any team of two strong engineers could build "chapters for diffs" in a quarter. The question I always ask is: if these founders disappeared tomorrow, would the company's core insight survive? Here, the insight is "organize diffs narratively instead of by file." That idea fits on a sticky note. It survives anyone's departure because it was never locked inside anyone's specific knowledge to begin with.

The specific knowledge assessment is where this deal falls short for me. Dean Stratakos built a coding agent at Five Rings and noticed that reviewing AI-generated code was painful. That's a real observation from lived experience — I won't dismiss it. But "code review is hard" is the most commonly expressed frustration in software engineering. Every engineer who's ever opened a 500-line pull request has felt this. Compare this to Aravind at Perplexity, who spent years inside Google Brain understanding the architecture of search at a level maybe fifty people on Earth shared. Dean's AI background — Stanford MS, Apple Siri rotations, the Five Rings agent work — gives him genuine facility with AI systems, but that knowledge points toward building AI tools, not toward a unique understanding of how human cognition processes code changes. The leap from "I noticed review is hard" to "I have a non-replicable insight about how to fix it" hasn't been demonstrated.

The leverage architecture concerns me more. Per-seat SaaS for developer teams is not permissionless leverage — it's the labor-intensive kind. You need team-level adoption, which means enterprise sales motion, onboarding, and constant API integration maintenance with GitHub. The "chapters" format has a modest viral quality within teams — one reviewer using Stage creates visibility for others — but that's table stakes for any collaboration tool. There's no data flywheel here. There's no network effect where user N+1 makes the product better for users 1 through N. And the platform dependency cuts deep: GitHub controls the API, the diff data, and the distribution surface. When Microsoft ships "AI-organized diffs" as a native Copilot feature — and they will, because Anthropic already launched multi-agent code review for Claude Enterprise in March 2026 — Stage's UX innovation becomes a feature, not a company.

Let me steel-man the bull case, because the timing argument is genuinely interesting. AI-generated code is creating a real comprehension crisis. The volume of changes per PR is exploding while the cognitive tools for review haven't evolved since GitHub launched pull requests. Graphite's acquisition by Cursor for reportedly >$290M validates that someone will pay serious money for better code review workflows — and Graphite's exit from the independent market creates an opening. If Stage captures the workflow before GitHub or Cursor fills the gap, and if "chapters" becomes the standard mental model for how teams think about code review (the way "stacked PRs" became Graphite's), there's a real lock-in dynamic. Two technical founders from Five Rings, both builders, shipping product together — that's the right team structure. The problem would need to be bigger and stickier than I think it is, and the "chapters" concept would need to generate a data advantage (learning to auto-structure diffs in ways that improve with scale) rather than remaining a static UX pattern. If Dean's AI expertise produced a system that genuinely learns how to decompose arbitrary changesets into semantically meaningful units — not just file grouping but actual intent reconstruction — that could be a real technical moat. I don't see evidence that's what's being built, but it's the path that would change my mind.

The principal-agent signal is mildly positive but not compelling. Both founders left presumably lucrative quant trading jobs, and the origin story — Dean built a coding agent, experienced the review problem firsthand, decided to solve it — is legitimate. They're not MBA graduates who identified a market opportunity through research. But the signal isn't strong enough to override the structural concerns. They met the problem through their work; they weren't consumed by it for years before founding. The co-founder relationship (Stanford classmates, Five Rings colleagues) is solid — no red flags there. Two builders who've worked together, both can ship code, both technical. That's necessary but not sufficient.

I pass. The core issue is that this is a UX innovation in search of a moat, competing against well-funded alternatives in a space where the dominant platform (GitHub) has every incentive and capability to absorb the innovation. Specific knowledge is generic, leverage is standard SaaS, and the "chapters" concept — while clever — doesn't compound. Innovation without a compounding advantage is a feature request, not a company.
