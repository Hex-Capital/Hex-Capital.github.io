# Glen -- Naval Ravikant Evaluation

Glen is a platform-shaped market bet on the agent-context layer executed through enterprise-gated distribution by a solo operator whose prior shipped artifact is a sales-coaching SaaS, not a retrieval system. The dimensional signature is a high mark on market shape — agent memory across MCP clients is genuinely platform-sized — and a low mark on leverage architecture, because Glen's distribution motion is "waitlist waves" with "work-email signup" reading from "code, PRs, issues, docs, and meetings" (tryglen.com, 2026). Read that sentence again: every source on that list requires per-customer enterprise authorization. That is permission leverage, not the permissionless kind I keep telling Nivi to fund.

The leverage architecture is where Glen diverges most from the **Twitter** (2007) template. Twitter was media at zero marginal cost — anyone posts, the platform charges nothing, replication is free. Glen's product, by its own description, reads from inside the corporate perimeter via OAuth 2.1 and a Postgres-backed encrypted store, and the front door is a gated waitlist requiring a work email. Each new customer is an enterprise authorization project — security review, SOC 2 questions Glen does not yet have, per-source connector approval. The MCP integration with Claude, Cursor, and Codex (tryglen.com, 2026) is code-leverage on the *consumption* side, but the *acquisition* side is the inverse of Stack Overflow's developer-pull pattern. When the entry motion is "book a Calendly via glen.cal.com/general/glen-chat," I have already scored the architecture down.

**Perplexity** (2022) is the single-sided builder-tool I gave the exception to, so let me trace whether Glen earns the same waiver. The override required three of five gates clearing at top quartile — permissionless API distribution, retrieval-PhD specific knowledge in Aravind Srinivas, and pre-consensus category timing. Glen has category timing — MCP-as-standard is real, and the agent-memory category is two years from narrative crystallization. But criterion #1 is permission-gated for Glen, not permissionless, and criterion #3 is the load-bearing question. Nikos Dritsakos shipped SalesBop into a SellWell/FliteHouse acquisition and served as VP Tech & Product there. That is operator competence — it is not the multi-year obsessive depth that retrieval systems and temporal knowledge graphs demand against Zep's Graphiti or Mem0's vector-first SDK. Exception denied.

Counterfactually on the network dimension, Glen's "anything learned by one agent is learned by all" (tryglen.com, 2026) would have to compound *across* organizations, not inside them, for this dimension to flip a tier. Per-org reconciled context is a workspace product like **Notion** (2013) — valuable, but each customer's graph is economically independent of the next. The Nth Glen customer does not make the (N-1) prior customers' agents smarter, because access control is "applied per observation, automatically, at recall" (tryglen.com, 2026) — by design, learnings stay inside the tenant. That is the right security posture and the wrong network mechanic. The upside math here is single-tenant SaaS multiples, not the asymmetric **Uber** (2010) geometry.

On market pivot-room, Glen sits in the strongest position of any dimension. Agent context is platform-shaped — Glen could be memory, then work-graph, then developer-tool runtime, then enterprise search adjacent to **Glean**, then a developer-IDE plugin layer. Mem0 raised $24M with 48K GitHub stars going one direction (vector-first per-agent SDK), Letta raised $10M at ~$70M post going another (OS-style agent runtime), Zep went a third (temporal knowledge graph). The category is wide enough to absorb two or three founder pivots inside the seed runway, which is the **Venture Hacks** requirement. This is the one dimension where Glen scores like a company I would fund.

Parallel to **Notion** (2013) on the specific-knowledge axis: Ivan Zhao had been rebuilding personal productivity tools since adolescence — the dossier called it adolescent-onward depth. Nikos Dritsakos ran a growth agency at university, helped scale Bright, co-founded SalesBop with two others, sold it, ran product at the acquirer. That is a competent serial operator — a profile I would back inside a category I understood, but it is not the lived-the-problem-as-user signature that the builder-tool wedge requires. The dossier names no GitHub handle, no open-source memory or retrieval work predating the round, no public writing on agent context. The artifact predating the equity story is absent.

Forecast at 36 months: Glen acqui-hires into Mem0 or Letta at sub-$20M, or shuts down. The mechanism is the collision of solo-founder bandwidth against a platform-scope product surface (connectors plus MCP server plus access control plus storage) inside a category where two funded peers already hold developer mindshare. Solo founders ship one thing well; Glen's value proposition requires shipping five things adequately and selling them through enterprise security review. The math closes on acquihire, not independent scale.

The strongest bull case is that the funded peers' commitments are exactly Glen's opening — Mem0 is structurally vector-first and per-agent, Letta is committed to autonomous-agent runtime, Zep is single-agent temporal graphs. None of them is positioned as the org-wide reconciled context layer Glen claims, and MCP is the genuine "why now" that makes a small team plausible. Nikos has a clean prior acquisition, which makes the integrity leg of the Buffett trio read fine. That bull case stings, but it dies on the leverage architecture: Mem0 distributes through AWS Agent SDK and 48K GitHub stars — *permissionless code leverage* I would fund — while Glen distributes through enterprise OAuth and a waitlist. The verdict flips if Glen ships an open-source MCP server with organic GitHub traction predating any priced round, or if a named co-founder with retrieval/systems depth joins before Demo Day. Until then, pass.
