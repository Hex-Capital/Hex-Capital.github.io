BinBin He worked on Firecracker at AWS — the exact technology that smol machines' core dependency, libkrun, incorporates code from. This is the most analytically important fact in the dossier, and it cuts both ways. On one hand, it means smol machines is built by someone who genuinely understands hypervisor engineering at the kernel level. On the other hand, it means the founder is a domain expert building in his domain — the precise founder archetype I'm least excited about. When I backed Peter Beck at Rocket Lab, the signal was a self-taught engineer from New Zealand who didn't know what aerospace engineers said was impossible. "Ship software faster with portable, self contained virtual machines." That's smol machines' tagline, and it tells me everything I need to know about consequence magnitude. If smol machines fully succeeds — every developer on earth uses `.smolmachine` files, Docker is displaced, QEMU is forgotten — what changes? Developers get sub-200ms boot times instead of 45+ seconds. They get hypervisor isolation instead of shared-kernel containers. Software ships with fewer "works on my machine" bugs. This is genuinely useful. It is not consequential. Nobody's healthcare gets cheaper. No basic human need becomes accessible to a billion people who lacked it. The world with ubiquitous smol machines looks approximately identical to the world without it, except developers are somewhat less frustrated. I don't invest to make developers somewhat less frustrated. Docker itself — $393M+ raised, massive ecosystem adoption, defining a generation of deployment practices — never made it into my portfolio for exactly this reason. Developer tooling has natural consequence ceilings.

The strongest bull case an advocate would make for smol machines centers on AI agent sandboxing. Claude Code, Cursor, Codex — every AI coding agent needs to execute untrusted generated code in isolation. The dossier notes a GitHub Discussion (smolvm #3) explicitly exploring this use case. If smol machines becomes the standard local sandbox for AI agents the way Docker became the standard for deployment, it's infrastructure underpinning the most consequential technology shift since the internet. The 2,500 GitHub stars in weeks suggest real developer pull. The timing argument is genuine: libkrun's maturity and AI agent proliferation create a window that didn't exist two years ago. I take this seriously. But I dismantle it on two grounds. First, E2B already has 88% Fortune 100 adoption for AI agent sandboxes with $21M in funding — smol machines is entering a use case where a funded competitor has massive traction. Second, even as AI agent infrastructure, smol machines is plumbing, not the revolution. The consequence explosion of AI comes from the model builders — OpenAI, Anthropic — not from the sandbox layer. When I wrote my $50M check to OpenAI, I wasn't investing in GPU orchestration tooling. I was investing in artificial general intelligence. Plumbing companies become Docker-sized ($2B). They don't become civilization-altering.

smol machines' 225x speed improvement over QEMU sounds like the kind of non-linear leap I look for — until you examine where that improvement comes from. The core VMM is libkrun, maintained by Red Hat's containers organization, incorporating code from Firecracker, rust-vmm, and Cloud Hypervisor. smol machines didn't create this technology discontinuity; Red Hat did. What smol machines built is a well-crafted Rust CLI (82.8% Rust codebase), a packaging format (`.smolmachine` files), and UX polish around an existing open-source VMM. This is integration engineering, not fundamental innovation. The dossier itself flags this: "Any team with deep systems programming expertise could build a similar CLI wrapper." When I funded Impossible Foods, the heme protein synthesis was proprietary science that no competitor could replicate without years of biochemistry work. When I funded Commonwealth Fusion Systems, the high-temperature superconducting magnets were a genuine physics breakthrough. smol machines' differentiation is UX and packaging on top of someone else's hypervisor library. That's a 2x improvement in developer experience, not a 10x discontinuity in technology.
