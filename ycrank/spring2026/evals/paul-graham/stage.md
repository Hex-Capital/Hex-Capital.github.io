# Stage -- Paul Graham Evaluation

The first thing I notice about Stage is the competitive landscape, and what it tells me about the problem's schlep depth. When I wrote about schlep blindness, the core insight was that the best startup ideas sit in plain sight for years because the unconscious mind filters out problems involving painful, tedious work. Stripe could exist because thousands of programmers knew payments were broken but nobody wanted to deal with banks, compliance, and fraud. The test for schlep blindness is simple: if the problem is real and obvious, why hasn't anyone solved it? With Stage, the answer is clear -- lots of people are trying. CodeRabbit has $88M in funding. Greptile raised $29M. Haystack, one batch earlier at YC, is pursuing nearly the same problem with a different visual metaphor. Graphite built a code review workflow tool used by 500+ companies before Cursor acquired it. When five funded competitors already exist in your space, the schlep barrier isn't keeping people out. This is a crowded room, not a hidden one.

That said, the origin story has real signal. Dean built an in-house coding agent at Five Rings, a quant trading firm with extremely high engineering standards, and that experience directly inspired Stage. He wasn't reading TechCrunch and deciding AI code review was hot -- he was living inside the problem, watching AI-generated code overwhelm the review process at a firm where code quality is existential. Charles worked across multiple engineering environments where review friction compounds. This is genuine organic discovery, the kind I wrote about in "How to Get Startup Ideas" -- the idea grew from the founders' own work, not from a brainstorming session. The timing insight is also real: AI coding tools are producing unprecedented volumes of unfamiliar code, and the review bottleneck is worsening faster than most people appreciate. Dean saw this before the broader market did because he was building the thing causing the bottleneck.

The founders themselves are technically excellent. Both Stanford CS, both from Five Rings -- which is one of those firms that's genuinely hard to get into, not just prestigious-sounding. Dean has a Stanford MS in AI and built production systems in C++ and Python. Charles did robotics at the Stanford AI Lab. These are hackers who can build things. The varsity athletics backgrounds (Dean was a Pac-12 tennis champion, Charles fenced saber) suggest the kind of competitive determination that matters -- not because athletics predicts startup success, but because sustained elite performance in anything requires the ability to push through pain. If I were evaluating them purely as founders, detached from this specific idea, I'd be interested.

Here's the bull case I keep coming back to: the "comprehension crisis" in code review is genuinely new and worsening. Before AI coding tools, a reviewer could reasonably assume the author understood their own changes. That assumption is breaking down fast. The founders saw this early because of Dean's agent work at Five Rings. If Stage's "structured chapters" become the default way teams process AI-generated diffs, the workflow lock-in could be powerful -- teams that adopt chapter-based review would find it very hard to go back to raw diffs, much like teams that used Graphite's stacked PRs found it hard to return to monolithic PRs. The Graphite acquisition also removed the closest independent competitor, opening a genuine window. Two disciplined hackers with low burn and deep technical chops could potentially capture this moment.

But I keep stumbling on the same problem: "structured chapters" reads like a feature, not a company. The dossier itself says it plainly -- this is "a UX innovation, not a deep technical moat." GitHub could add logical grouping to its diff view. Cursor, which now owns Graphite, could implement chapters as a feature of their expanding developer platform. When your core innovation is a way of organizing information visually, you're in a race against every platform that displays the same underlying data. Dropbox had this problem too -- "anyone could build file sync" -- but Dropbox had the advantage that the schlep of building reliable cross-platform sync was harder than it looked. I'm not convinced that building structured chapters has that same hidden depth. The concept is elegant. But elegance alone doesn't create defensibility.

The complete absence of traction evidence concerns me less at pre-seed than the structural question of whether this is a feature or a product. I don't penalize companies for not having users before they've launched. But I do want to see some signal -- a handful of teams using a prototype, a waitlist, engineers sharing excitement on Twitter -- that suggests the product creates the kind of reaction where people tell their teammates unprompted. There's nothing here. Not even a public demo. For a tool targeting developers, where bottom-up adoption is the only viable distribution channel for a two-person team, the absence of any visible community engagement or early user signal is a gap. It doesn't condemn, but it doesn't inspire.

This is a competent team working on a real problem with weak schlep and unclear defensibility. The organic discovery is present but the problem itself is too visible, too acknowledged, and too actively pursued by well-funded competitors for me to see the structural advantage that turns a good idea into a great company. I'd want to see evidence that the "chapters" concept creates genuine workflow lock-in -- that teams who try it can't go back -- before writing a personal check.
