# Sparkles -- Sam Lessin Evaluation

Here's the thing about Sparkles that activates every alarm in my framework simultaneously: this is a software product whose entire existence depends on AI, built as a layer on top of someone else's platform (GitHub), using someone else's models (third-party LLMs), targeting a use case that every well-funded incumbent in the space could add as a feature toggle. "Lovable for existing projects" is not a company — it's a checkbox on Lovable's product roadmap. When I say software is a business tool and not a business model, Sparkles is what I'm describing. Strip away the AI and there is literally no product. This isn't "cherry on top" — this is just cherry. No sundae underneath.

The sane-person-insane-idea filter produces an unusual result here: a reasonably credible young founder pursuing a completely consensus idea. Daniil Bekirov has relevant exposure — Cursor Ambassador, engineering at a YC company, intern at Iterate — which gives him genuine feel for the developer tools space. That's the sane part. But "let non-technical people edit code using AI" is the single most obvious application of large language models that anyone could imagine. Every pitch deck in this category leads with the same insight: most people in an organization can't write code, AI can bridge that gap. When every VC in the room nods along, you're not investing in a non-consensus idea — you're buying narrative at market price. The best version of my framework is a sane person doing something most people think is crazy. This is a sane person doing something most people think is obvious, which is the quadrant where returns go to die.

Run the AI kill test and the answer is immediate: AI doesn't strengthen this business, AI IS this business. Now run it in the other direction — does improving AI help or hurt Sparkles? As models get better, the gap between "editing an existing codebase" and "generating a new one that matches your existing one" collapses. Better models make Lovable, Bolt.new, and Cursor more capable of handling existing repos, which is exactly the narrow wedge Sparkles occupies. The competitive landscape is genuinely brutal — Lovable at $6.6 billion, Cursor at $29.3 billion — and the structural barrier the dossier identifies (that these tools are architecturally built for greenfield projects) is an engineering constraint, not a moat. Engineering constraints get solved with engineering effort, and these companies have billions in capital to deploy.

The strongest bull case would require Sparkles to become a new primitive — the standard organizational interface through which non-engineers interact with production code, accumulating enough company-specific context and editing patterns to create real switching costs. If every marketing team, every ops team, every design team at a mid-market company defaulted to Sparkles the way they default to Figma for design, that's a genuine platform position. But Figma had a decade of pre-AI development during which its collaboration model compounded into infrastructure-grade lock-in. Sparkles is starting in a world where the underlying capability — AI code generation — improves faster than any application layer can build defensibility around it. The window to establish that primitive position before incumbents absorb the feature is measured in months, and you're asking a solo 20-year-old founder to out-execute companies with thousands of engineers and billions in ARR. I've seen that movie before, and the solo founder doesn't usually win the distribution race.

Solo founder risk compounds everything. Building web sandboxing, GitHub integration, enterprise features, and AI code generation simultaneously as a team of one — with one additional GitHub contributor visible — is a concentration risk that makes the already-narrow execution window even narrower. The angel backing from individuals at OpenAI, xAI, a16z, and others provides network access but not the operational bandwidth this product needs to reach escape velocity before Lovable ships "import repo" as a feature.

I keep coming back to Birchbox as the structural analog. A real insight about a real gap in the market, pursued by credible people, but with a moat that was trivially replicable once established players noticed the opportunity. Sparkles has identified a genuine user pain point — non-engineers struggling with git and terminal to make simple front-end changes. That pain is real. But the solution is a product feature, not a company. And in a world where software moats dissolve faster than they form, building a pure-software product with no physical-world operations, no protocol-layer position, and complete dependency on third-party AI models is precisely the bet I'm structured to avoid.
