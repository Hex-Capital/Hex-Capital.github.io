# Sonarly -- Vinod Khosla Evaluation

Here's what strikes me immediately: Sentry already reports 95% root cause accuracy and 54% fix rate on its own error data. Datadog's Bits AI Dev Agent has been tested across 2,000+ customer environments. Two public companies with billions of dollars in market cap and massive developer distribution are shipping the exact capability Sonarly is building — as features of their existing platforms. Sonarly's structural argument is that these incumbents can't aggregate across each other's data because it would cannibalize their lock-in business models. That's an astute observation about market structure. But it raises a question I find more important: even if you win the cross-platform aggregation game, does the world change?

If Sonarly succeeds — I mean really succeeds, dominates the category — the consequence is that engineering teams spend less time triaging production alerts and more time writing features. That's a genuine productivity gain. But it's exactly the kind of outcome I warn founders about. You've built a better workflow tool. Nobody's life is fundamentally different. No trillion-dollar system becomes obsolete. Sentry still exists. Datadog still exists. The monitoring stack still works the same way. Sonarly sits in the middle as middleware, aggregating and automating what engineers used to do manually. A user in the Hacker News comments reported going from 180 alerts to 5 actionable issues — impressive operationally, but this is optimization, not transformation. The consequence of success here is a good developer tools business, not an industry restructuring. I've seen hundreds of companies that make some enterprise workflow 30% more efficient. The team is capable, the market exists, the technology works. And I keep asking the same question: if this succeeds, so what?

The strongest bull case would require believing that "AI that fixes production" is a wedge into something far larger — an autonomous software maintenance layer that fundamentally changes how software is built, where human engineers write new code and machines maintain all existing code. That vision, if you extend it far enough, has real consequence magnitude. But the dossier describes a triage and deduplication layer with a 30% mergeable PR rate, not an autonomous SRE. The gap between "filters your Sentry alerts" and "replaces your on-call engineering team" is enormous, and the founders haven't articulated the larger vision in any public source I can evaluate. If they privately believe they're building the autonomous maintenance layer for all production software — that would be a different conversation. But what I see is a lightweight integration layer solving alert fatigue, and that's calibrated far below the scale of problem I look for.

The technology itself is assembled from generally available components — LLM-driven code analysis, monitoring API integrations, MCP protocol. There's no proprietary breakthrough here. The "living map" of production systems stored as a .md file is an interesting concept for building switching costs, but it's unproven and any competitor with API access to the same monitoring tools could build equivalent context. When Sentry has deeper access to its own error data and already achieves 54% automated fix rates, a cross-platform aggregator with shallower access to each tool may actually perform worse per-source, even if it sees more broadly. The technology doesn't make incumbents structurally obsolete — it depends on their continued existence.

On the founders: Dimittri and Alexandre have a multi-year working relationship from CY Tech and their prior edtech company MEORIA, and they discovered this problem from genuine personal pain — 50 daily alerts on a 2-person team. That founder-problem fit is real. But building an AI-powered alert triage tool in 2026 is not contrarian. It's consensus. Every monitoring vendor is doing this. The founders aren't outsiders challenging what experts consider impossible — they're developers building what every developer wants. I don't see evidence of the kind of contrarian courage or rapid learning-rate evolution that characterizes my best investments. Neither founder has infrastructure or developer-tools company experience, which for an outsider attacking a transformational problem could be an advantage, but for a team building incremental tooling in a competitive market, it's a gap.

The timing trajectory is genuinely favorable in one dimension: LLM code-generation quality is on an exponential curve, and MCP standardization reduces integration costs. But the timing is unfavorable in another: the incumbents are already here. This isn't 2019 AI where I could see the trajectory before anyone else. This is 2026 AI-powered developer tools, where Sentry, Datadog, and well-funded startups like incident.io ($96M raised) are all converging on the same space simultaneously. The founders aren't skating to where the puck will be — they're arriving at the same spot as everyone else.
