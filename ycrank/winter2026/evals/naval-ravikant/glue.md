# Glue -- Naval Ravikant Evaluation

The competitive landscape tells the story before the founders do. Cursor is at $2B+ annualized revenue and a $29.3B valuation. Lovable crossed $200M ARR. Vercel, with v0, has $563M in funding. Figma is valued at $12.5B. Glue is proposing to build a visual design canvas for coding agents in a field where the well-funded incumbents have every incentive and every resource to add this exact feature. When I backed Uber at $4M, the incumbents — taxi companies — couldn't build what Travis was building because the technology was alien to their organizations. Here, the incumbents are AI-native engineering companies. Cursor adding a visual canvas is not a category leap — it's a product sprint. The structural advantage that makes an early bet worthwhile in a crowded space — some barrier that money alone can't overcome — I don't see it.

The specific knowledge question is where this falls apart for me. Perbhat Kumar redesigned the Kindle Fire launcher and built RAG agents at Copilot Studio. Tejas Priyadarshi was an engineer at Meta and worked on Azure DNS. These are solid resumes. They are not specific knowledge. Specific knowledge is what Evan Williams knew about public communication after years building Blogger — understanding that couldn't be extracted from him and poured into a competitor. What do these founders know about the design-to-code workflow for AI agents that a sharp engineer at Cursor or Vercel couldn't learn in a quarter? The dossier reveals no published thinking, no years-long obsession with this intersection, no contrarian insight about why existing tools are wrong at the architectural level. The CTO's GitHub bio still references "Willow (YC W26)," which suggests a recent pivot — meaning the current product may be weeks old, not the product of years of accumulated conviction. When the product is new and the knowledge is transferable, you're in a sprint against better-capitalized teams, and sprints against billion-dollar companies are not where angel returns come from.

The leverage architecture is the second structural problem. Glue is not creating a new form of leverage — it's building a user interface on top of leverage that already exists. The coding agents themselves (Claude Code, Codex) are the leverage layer. They turn one person's intent into production code. Glue adds a visual canvas for that interaction, which is useful but not leverage-creating. Compare this to Stack Overflow, which built a knowledge graph that compounded with every question and answer, creating leverage that didn't exist before. Or Replit, which made programming accessible to people who couldn't previously code — a qualitatively new form of leverage. Glue makes an existing workflow slightly more visual. That's an application, not infrastructure. My portfolio has always tilted toward the infrastructure layer because infrastructure captures value permanently while applications get competed away.

The bull case is worth articulating. If design-for-agents becomes a genuine category — if the way humans communicate design intent to AI agents is a fundamental unsolved problem — then whoever defines the canonical visual surface wins a standards war. Figma can't easily pivot because its architecture optimizes for designer-to-designer handoff, not agent-to-builder collaboration. If Glue ships fast, open-sources effectively, and becomes the surface that agents are fine-tuned to interact with, the network effects could be powerful: more agent integrations attract more builders, more builders create more component libraries, more libraries make the platform stickier. The timing is arguably right — coding agents reached production quality in 2024-2025, and the "vibe coding" movement has created a new builder persona that existing tools don't serve. If you squint, this is the kind of moment where a small team with the right product could define a category before the big players realize it exists. But for that bull case to hold, the founders would need to ship the open-source repo (which doesn't exist publicly), build a community before competitors notice, and demonstrate that a standalone visual canvas has enough gravity to resist being absorbed as a feature. None of those conditions are currently met.

The open-source strategy is stated but not activated — no public repository was found despite the company describing itself as open source. Open-source distribution is powerful precisely because it creates community investment before monetization. GitLab and Supabase built real communities before building businesses. Claiming open source without a repo is like claiming network effects without users. It's a strategy document, not a strategy.

I'd pass. Competent founders in a real market, but the knowledge is transferable, the leverage is derivative, the category is peak consensus, and the technology doesn't compound in ways that create durable separation from teams with a hundred times more capital. The founders are building an application in a space where I'd want to own the infrastructure.
