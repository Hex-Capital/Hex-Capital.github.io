# Stage -- Elad Gil Evaluation

The competitive dynamics here tell you almost everything you need to know about the structural position of this company. CodeRabbit has raised $88M and is doing $15M+ ARR. Greptile raised $29M with Benchmark leading. Graphite was acquired by Cursor for reportedly over $290M. And Haystack, a YC S24 company pursuing nearly the identical problem -- making diffs comprehensible for human reviewers -- has a full year's head start. Stage is entering a space where well-capitalized players have already staked out the territory, and the closest direct competitor is a YC portfolio sibling one batch ahead. When I see this kind of landscape, my first question isn't "is the product better?" -- it's "what structural shift gives a new entrant a window that the existing players can't exploit?" And I'm struggling to identify one.

The "why now" thesis is that AI code generation is producing volumes of unfamiliar code that overwhelm human reviewers, creating a comprehension crisis. This is real -- I've been tracking this dynamic across my portfolio -- but it's completely obvious. Every developer tools investor in the market sees this trend. The more interesting contrarian bet embedded in Stage's approach is that the right response to AI-generated code is restructuring human comprehension, not automating review with more AI. Most capital is flowing toward AI review bots (CodeRabbit, Greptile, Cubic), and Stage is betting the bottleneck is the human reviewer's cognitive load, not the absence of automated flagging. That's a defensible product intuition. But a product intuition is not a structural market opening. When I backed Harvey, the nonobvious element was that a specific AI capability threshold was being crossed that would unlock a market stuck for a decade. Here, the underlying capability -- organizing diffs into logical groups -- has been technically feasible for years. Nothing about the current moment makes "structured chapters" newly possible in the way GPT-4 made legal document analysis newly possible.

The infrastructure-layer test is where this falls apart for me. Stage is a UX application layered on top of GitHub's API. It doesn't build a platform that other businesses integrate into. There's no API-first architecture, no data layer that compounds, no developer ecosystem building on top of Stage. Compare this to Stripe, which became the payments infrastructure that thousands of companies built on, or even to Graphite before its acquisition, which at least introduced a workflow primitive (stacked PRs) that was structurally impossible in GitHub's native UI. "Structured chapters" is an organizational metaphor -- it reorganizes information that GitHub already exposes. GitHub could add a "group changes by logical unit" feature in a single sprint. The dossier itself flags this: "The 'structured chapters' concept is a UX innovation, not a deep technical moat." When I'm evaluating picks-and-shovels opportunities during a platform shift, I need the company to BE the platform layer, not to sit on top of someone else's. Stage has the opposite positioning -- it's dependent on GitHub's API, which means GitHub has the option value to subsume the feature set at any time.

The single-miracle test reveals at least two independent things that need to go right. First, the chapter-based UX needs to be dramatically superior to both GitHub's native review and Haystack's canvas approach -- a product/UX miracle. Second, engineering teams need to adopt a standalone review platform that lives outside their existing GitHub workflow -- a distribution miracle. Graphite succeeded at the distribution challenge partly because stacked PRs were structurally impossible in GitHub's native UI, giving developers a reason to leave the default. But reorganizing a diff into chapters doesn't require leaving GitHub -- it's an enhancement to an existing flow, which means the switching motivation is weaker and the risk of GitHub commoditizing the feature is higher. Two miracles, compounding odds.

The bull case I'd want to stress-test: if Stage nails the UX and "chapters" becomes the default mental model for how engineers review AI-generated code, this could become deeply habitual -- the way Figma's multiplayer canvas became the default design workflow. Workflow habits create real switching costs even without technical lock-in. Dean Stratakos's background is relevant here: Stanford MS in AI, led the AI initiative at Five Rings, built an in-house coding agent that directly inspired this product. If anyone understands how AI-generated code differs from human-written code and what reviewers actually need, it's someone who built the agent producing that code. And the Graphite acquisition by Cursor genuinely did remove the leading independent review workflow tool, creating a gap. If Stage captures the comprehension-focused segment while everyone else chases AI automation, and if the chapter metaphor proves sticky enough to resist commoditization, this could work. But that's a lot of "ifs," and the moat remains thin -- I've seen too many UX-only innovations get absorbed by incumbents once the category proves valuable.

The founders are technically strong -- Stanford CS, quant trading at Five Rings, relevant AI experience -- but I don't see scaling signals. Neither has operated at a company during hypergrowth or navigated the organizational challenges I write about in the High Growth Handbook. They're first-time founders building in a space with well-funded competitors who have multi-year head starts. At pre-seed I don't penalize this heavily, but it doesn't add conviction either. The per-seat SaaS model that the dossier infers is exactly the pricing architecture my "seats to cognition" framework warns against -- as AI reduces the number of engineers needed per project, per-seat pricing compresses. I'd want to see outcome-based or usage-based pricing (per PR reviewed, per review cycle saved) to believe in margin expansion over time.
