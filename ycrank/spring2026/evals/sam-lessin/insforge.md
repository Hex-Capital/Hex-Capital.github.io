# InsForge -- Sam Lessin Evaluation

Supabase has $500M+ in funding, 80,000+ GitHub stars, and has already shipped MCP support for AI agent interaction. InsForge has $2M, 7,700 stars, and a "semantic layer" that does the same thing Supabase is building as a feature. This is the whole ballgame. InsForge is betting that the interface between AI coding agents and backend primitives — Postgres, auth, storage, edge functions — is so architecturally distinct that it warrants a new platform rather than a new API endpoint on an existing one. I've seen this movie before. Every time you get pitched "incumbent product, but optimized for [new paradigm]," you have to ask: can the incumbent just add that optimization? Supabase is a Postgres wrapper with APIs. Adding an agent-optimized API layer is not a pivot — it's a sprint ticket. InsForge's core differentiation is a software abstraction over commodity primitives, and the $5B-valued competitor has already started replicating it.

InsForge sits dead center in the hottest consensus category in venture right now — AI developer tools. Half the YC Spring 2026 batch is probably pitching some variation of "infrastructure that AI agents can use." No one is dismissing this category. No one thinks this is uncomfortable or crazy. The market is racing to fund exactly this thesis, which means any upside is already baked into the price. When I backed Solana at four cents, every smart crypto person had written off alternative L1s — the consensus was wrong, and the price reflected the consensus, not reality. InsForge is the opposite: the consensus is that AI-native dev tools are the future, VCs are tripping over each other to fund them, and InsForge is priced as if the category risk is real when it's actually the execution risk against a massively capitalized incumbent that's real. There's no mispricing here — there's overpricing of the entire category.

Hang Huang and Tony Chang are competent people building a sensible product. Hang has a Yale MBA and Amazon experience; Tony has Databricks and Meta infrastructure engineering with 10 hackathon wins. The six YC applications before acceptance shows persistence, which I respect. But "sane person, insane idea" requires both halves. The idea — making Supabase's backend primitives more consumable by AI agents — is not insane. It's obvious. Everyone in the developer tools space is asking the same question InsForge is answering. When I get excited, it's because a rational founder is telling me something most smart people think is impossible. "Backend infrastructure should be agent-native" is something every smart person already agrees with. InsForge isn't contrarian; it's early to consensus, which is a very different value proposition.

The bull case writes itself, and I want to be honest about it: InsForge's 4,000+ hosted databases, #1 Product Hunt launch, and #1 GitHub Trending suggest real developer pull. The self-reported MCPMark benchmarks — 1.6x faster task completion, 1.7x higher accuracy versus Supabase — could indicate genuine architectural advantage in agent consumption. If AI agents become the primary interface for all backend provisioning within 3 years, and if InsForge's semantic layer creates enough switching cost through production database lock-in, they could build the new default the way Supabase became the default open-source Firebase. The Slack exception in my framework says I'll pay consensus prices when the primitive is real. But InsForge isn't creating a primitive — Supabase already created the open-source BaaS primitive. InsForge is adding an interface layer optimized for a new type of consumer. That's an incremental UX adaptation, not a Venmo-style behavioral invention. Venmo created a verb; InsForge is creating a configuration file format.

InsForge's structural durability problem runs deeper than the typical AI kill test. The standard question is "can an incumbent add AI to replicate this?" — but InsForge's problem is inverted. InsForge IS the AI optimization layer, and the incumbent doesn't need AI to kill it; Supabase just needs to ship better agent APIs, which it's already doing with MCP support. The semantic layer and `skill.md` workflow reference are software abstractions with no physical moat, no protocol-level network effect, and no proprietary data flywheel. Apache 2.0 licensing means the community itself isn't defensible — anyone can fork it. The $300/mo revenue at InsForge's last disclosed data point, combined with a generous free tier (50K MAU for $0/mo), suggests a long and uncertain conversion path. InsForge would need to demonstrate that agent-native architecture requires fundamentally different infrastructure decisions at the database and compute layer — not just a different API surface — for the structural moat to be real.

One condition would flip me: if InsForge's benchmarks were independently validated and showed that agent performance degrades so substantially on Supabase's bolted-on MCP support that production teams cannot use Supabase for agent workflows — proof that the architectural difference is structural, not cosmetic. That would transform this from "better interface" to "different infrastructure," which is a meaningfully different bet. Without that evidence, InsForge is a feature competing with a platform. The two dimensions that lock this verdict are Consensus Mispricing (AI dev tools is the most consensus-hot category in YC right now, zero contrarian edge) and Structural Durability (Supabase's convergence via MCP support is existential, and the semantic layer is a software abstraction over commodity primitives). Pass.
