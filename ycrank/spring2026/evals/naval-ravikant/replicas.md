# Replicas -- Naval Ravikant Evaluation

The first thing I see here is a leverage architecture built on someone else's foundation. Replicas orchestrates Claude Code and Codex inside cloud VMs. But the actual leverage — the AI doing the coding — belongs to Anthropic and OpenAI. This is the fundamental problem. When I backed Notion, the product WAS the leverage. When I backed Perplexity, Srinivas owned the intelligence layer. Replicas is building a dispatch surface for leverage that lives elsewhere. The moment Anthropic ships native cloud execution for Claude Code — or OpenAI extends what Codex already has — the orchestration layer compresses to a feature. OpenAI has already done this. That's not a hypothetical competitive risk; it's a present-tense architectural fact.

Now, the specific knowledge question. Connor Loi spent eight months on the core team at Helicone building an AI Gateway and LLM observability tooling. That's relevant operational experience — he understands LLM request routing and developer infrastructure plumbing. But eight months of adjacency is not the same as years of obsessive engagement that would take a competitor a decade to replicate. If Connor disappeared tomorrow, could a competent infrastructure engineer rebuild this with the same three integrations (Slack, Linear, GitHub) in a few months? Probably yes. The specific knowledge here is useful but transferable. Compare to Ev Williams, who had years of irreplaceable knowledge from Blogger about how public communication actually worked online. Connor knows how to route LLM traffic — that's infrastructure skill, not a proprietary understanding of how engineering teams will fundamentally reorganize around AI agents.

The principal-agent dynamics here actually work against Replicas, which is unusual for a startup. Normally I look for startups where the incumbent is structurally unable to compete — Google's ad revenue model prevents them from building honest search, so Perplexity wins. But Anthropic and OpenAI have every incentive to make their agents run seamlessly in the cloud. They're not agents trapped by legacy revenue; they're principals building the future of software development. There's no structural inhibition preventing them from eating this layer. The "agent-agnostic" positioning is the best argument for survival — if multiple coding agents remain competitive, a neutral orchestration platform has a role. But neutrality alone is a thin moat. It's the Switzerland argument, and Switzerland works in geopolitics but not in software.

The bull case deserves serious consideration. The timing is genuinely right — Claude Code and Codex just became viable headless agents in the last year, and the shift from synchronous to asynchronous AI coding is real. There's a window where no one owns the dispatch layer. If Connor moves fast enough and the workflow integrations (Slack, Linear, GitHub) become genuinely sticky — if teams start thinking "assign this ticket to Replicas" the way they think "push to GitHub" — then the dispatch surface itself becomes the product. The 30% PR completion rate, if real across multiple teams, suggests the product actually works. Three named customers with a V1 from a solo founder is honest traction. The Helicone connection is a network advantage, not a weakness — Connor saw this problem from the inside of LLM infrastructure. If I squint, this looks like early-stage AngelList — a tool that routes demand to the right supply in a new market. But AngelList worked because I had years of specific knowledge about what the angel market actually needed, and the supply (startups) couldn't be commoditized. Here, the supply (cloud VMs running CLI agents) is the definition of a commodity.

One person shipping a working product with real customers — that I respect. Solo founder, student, no prior founding experience, and he's got three engineering teams using the product. That's the builder signal I care about. But the GitHub portfolio tells me he's a broad builder, not an obsessive specialist. Ray tracers, space games, language apps, PR stacking tools — he builds everything, which means this particular problem might not be the one where his specific knowledge compounds over time. The solo founder structure also means one person doing product, engineering, sales, infrastructure operations, and support. At pre-seed this is fine. At scale, it's a constraint — and cloud infrastructure has real operational burden.

I pass. The core issue is structural: the leverage is rented, the orchestration layer is thin, and the incumbents have both the incentive and the capability to subsume this. Connor is a capable builder — I'd want to see what he builds next, particularly if he goes deeper into a problem where his specific knowledge is the product rather than a convenience wrapper around someone else's intelligence.
