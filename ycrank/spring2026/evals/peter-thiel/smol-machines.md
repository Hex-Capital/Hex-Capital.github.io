# smol machines -- Peter Thiel Evaluation

BinBin He built smolvm on top of libkrun — a VMM maintained by Red Hat's containers organization — and therein lies the structural problem. smol machines has constructed an elegant engineering layer on a foundation it does not own. Docker made this exact mistake in reverse: it created the container category, achieved massive developer adoption, and then watched as AWS, Google, and Microsoft absorbed container orchestration into their platforms while Docker struggled to capture monopoly value. Docker at least controlled its own runtime and image format with years of ecosystem lock-in. smol machines controls neither its VMM layer nor any meaningful artifact ecosystem — 2,500 GitHub stars and a `.smolmachine` format with zero third-party tooling do not constitute switching costs. The dossier states plainly that "any team with deep systems programming expertise could build a similar CLI wrapper." When the core technology is open-source and the integration layer is replicable, you do not have a secret — you have a feature that an incumbent will ship as a checkbox.

smol machines' implicit contrarian claim — that developers need hypervisor-level isolation with sub-200ms boot times on local machines, not shared-kernel containers — is partially correct but not a monopoly-creating insight. Firecracker already proved microVMs work; AWS runs Lambda and Fargate on them. The engineering achievement of bringing this to macOS desktops via Apple's Hypervisor.framework is genuine, and BinBin He's prior Firecracker experience at AWS gives him authentic domain credibility. But "bring server-side technology to developer laptops" is an execution insight, not a secret about the world that most investors would reject. E2B raised $21M for AI agent sandboxes using Firecracker. Fly.io operates Firecracker-based machines at $115M in funding. Google's gVisor provides similar isolation guarantees. smol machines enters a landscape where multiple funded players pursue overlapping theses — the competitive landscape slide that I treat as a disqualifying signal practically writes itself. Competition is not an obstacle to overcome through superior execution; it is a structural condition that destroys value for all participants.

The AI agent sandboxing angle — Claude Code, Codex, and other coding agents needing isolated execution environments for untrusted generated code — is the strongest demand signal in smol machines' favor. A GitHub discussion (issue #3) explicitly references this use case. If smol machines became the standard local sandbox for AI agent code execution, the TAM would expand dramatically beyond traditional developer tooling. But this thesis is pure consensus: every developer infrastructure investor sees the AI agent sandbox opportunity. E2B has already captured 88% of the Fortune 100 for cloud-hosted AI sandboxes. The question is not whether the demand exists — it does — but whether smol machines possesses any structural advantage in capturing it. An open-source CLI tool with no commercial layer, no pricing page, and no stated monetization plan does not describe a company positioning itself to monopolize this opportunity. It describes a project that hopes adoption will eventually generate a business model — the indefinite thinking I explicitly oppose.

The bull case deserves construction before dismissal. BinBin He is not a random developer — he worked on Firecracker at AWS, the production system that powers Lambda, and he is building smolvm in Rust with 476 commits and genuine systems-level depth. The `.smolmachine` artifact format could become a distribution standard for shipping sandboxed applications, analogous to how Docker images became the universal packaging format. The "Electron for VMs" analogy — shipping an entire isolated VM with your software — addresses a real pain point for CLI and desktop software distribution. If smol machines captured this format war, network effects around the artifact ecosystem could create genuine lock-in. And the timing is legitimate: libkrun's maturation, Apple Silicon's hypervisor support, and AI agent proliferation converge to make this product possible now when it was not two years ago. I have made exceptions before — Lyft violated "competition is for losers" because the ride-sharing category generated enough total value that even the #2 player produced fund-relevant returns. Does smol machines merit a similar exception?

No. The Lyft exception applied because ride-sharing was creating hundreds of billions in new value and even the loser would be worth $20 billion. smol machines operates in developer tooling, where the entire Docker company — after a decade of category creation and universal adoption — achieved a valuation that would not return a meaningful fund. The power-law math does not work. smol machines at maximum success becomes a useful developer tool worth perhaps a few hundred million dollars — a "nice business" that is structurally irrelevant to fund returns. The `.smolmachine` format war against Docker images would require overcoming network effects that compound annually across millions of developers, CI/CD pipelines, and cloud registries. Fighting that war with a team the dossier lists as size 1, no revenue, and no commercial product is not definite optimism — it is undercapitalized ambition in a competitive market. Fu Qiao's background is entirely undocumented, and the co-founders show no identifiable shared prehistory — a founding relationship I cannot evaluate is one I cannot trust with conviction capital.

smol machines is a pure software abstraction layer with no physical-world moat, no government customer pathway, no regulatory barrier, and no proprietary technology beneath the UX. Compare this to General Matter, where I took a board seat: proprietary HALEU enrichment technology, a $900M DOE contract creating structural switching costs, and physical infrastructure that no competitor can replicate with venture capital alone. Or compare to Palantir's earliest stage: even before revenue, the CIA's In-Q-Tel backing signaled institutional embedding that would compound into operational indispensability. smol machines has none of these monopoly mechanisms. The one condition that would change my assessment: if smol machines secured an exclusive integration as the default sandboxing layer for a major AI platform — becoming the Palantir of AI agent execution, operationally embedded and structurally irreplaceable. Without that, this is a well-executed open-source project in a competitive market with no path to monopoly. The decisive dimensions are Monopoly Secret and Power Law Magnitude — smol machines lacks a non-obvious insight that creates structural dominance, and the realistic ceiling on outcomes falls below the threshold where the power law makes a single investment matter.
