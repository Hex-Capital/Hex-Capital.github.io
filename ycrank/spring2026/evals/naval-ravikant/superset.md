# Superset -- Naval Ravikant Evaluation

The dependency arrow points the wrong way. I love infrastructure and picks-and-shovels plays -- Alchemy worked because Ethereum had no incentive to build a developer platform itself. CoinList worked because protocols needed a neutral distribution layer that no single blockchain would build. But Superset is an orchestration layer sitting *on top of* agents built by companies that are actively absorbing orchestration as a core feature. Claude Code shipped Agent Teams the same month Superset launched. Cursor shipped cloud agents the same week. This isn't a future risk to monitor -- it's the present competitive reality. Every agent provider has direct commercial incentive to make multi-agent orchestration native: it increases usage, which increases API revenue or subscription stickiness. Superset depends on the agents; the agents don't depend on Superset. That's the application layer pretending to be infrastructure, and the distinction matters enormously for durability.

On specific knowledge: three ex-YC CTOs, all technical, all capable builders. Kiet Ho demonstrated genuine open-source distribution ability with Onlook -- 23.4k GitHub stars is not an accident. That's a real, transferable skill. But what non-teachable insight about agent orchestration, multi-agent coordination, or the fundamental architecture of how AI systems should work together does this team possess? They left their W25 companies and built an Electron app that runs multiple terminal sessions in isolated Git worktrees. That's solid engineering executed quickly, not deep domain knowledge accumulated over years. The question I always ask is whether a team of three strong engineers from any top YC batch, given six months, could independently arrive at the same product. Here the answer is clearly yes -- and several have, which is why Multiclaude, Gas Town, and native agent orchestration features all emerged simultaneously. When everyone converges on the same solution at the same time, you're not looking at specific knowledge. You're looking at obvious timing.

The leverage test fails in a way that concerns me more than the competitive dynamics. Superset doesn't create a new form of leverage -- it marginally improves the efficiency of existing leverage (the AI agents themselves). Running ten agents in parallel instead of one sequentially is a productivity gain, not a leverage shift. Compare this to Replit, which took someone who couldn't code and gave them code as leverage -- a qualitative transformation in what a person could do. Superset takes someone already using Claude Code and lets them use it slightly more efficiently. There are no network effects in the product. No marketplace dynamics. No protocol layer. No compounding data advantage. It's a workflow tool, and workflow tools without switching costs get absorbed by the platforms they sit adjacent to.

The strongest bull case: the AI coding agent market fragments into many providers, no single agent wins, and developers use multiple agents the way they use multiple languages. In that world, an agent-agnostic orchestration layer becomes the VS Code of agents -- the neutral environment where all agents run. The 6,500 GitHub stars in weeks demonstrate real developer pull, and this team has proven twice over that they can build and distribute open-source tools that developers actually adopt. If the team moves fast enough, they could establish the default workflow before any single incumbent absorbs it. I take this seriously. But the analogy to VS Code breaks at the incentive layer -- programming languages don't build their own IDEs because they have no commercial incentive to do so. Agent providers absolutely have that incentive, and they're already executing on it. The window for an independent orchestration layer may be measured in months, not years, and building a durable business in a closing window requires either deep technological compounding or a monetization engine -- Superset has neither yet.

This is a talented technical team chasing a consensus wave without the specific knowledge or leverage architecture that would make it a compounding bet. The execution is fast, the distribution instincts are proven, but the structural position is fragile. I'd want to see evidence that these founders understand something about multi-agent coordination that the teams at Anthropic, OpenAI, and Cursor don't -- some architectural insight about how agents should communicate, share context, or resolve conflicts that goes beyond "run them in parallel in separate Git worktrees." Without that, this is a feature, not a company.
