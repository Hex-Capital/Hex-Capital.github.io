# smol machines -- Sam Lessin Evaluation

BinBin He worked at AWS on Firecracker — the actual technology that powers Lambda and Fargate — and then left to build smol machines on top of libkrun, an open-source VMM maintained by Red Hat. Let me be precise about what that means: smol machines' founder has deep, legitimate expertise in exactly this domain, and he's building on top of a virtualization layer he does not control, maintained by an organization (Red Hat/IBM) whose priorities could diverge at any moment. That tension — genuine domain mastery combined with structural dependency — is the most analytically honest thing about smol machines. The founder-market fit is real. The moat is not.

Docker raised $393M+. E2B just closed a $21M Series A with 88% Fortune 100 adoption. Fly.io sits at $115M raised and a $467M valuation. Developer infrastructure is one of the most consensus-hot categories in venture capital right now, and anything adjacent to "sandboxed execution for AI agents" has every seed investor on Sand Hill Road salivating. smol machines is operating in exactly the territory where the upside is already priced in. There's no discomfort here. Nobody's been told "no" because the category makes investors queasy — this is the opposite of Eaze in cannabis or my Solana bet when every crypto person said Ethereum had won. When I look at smol machines, I see a well-built developer tool in a category where ten other VCs are already bidding. The mispricing just isn't there.

smol machines' own tagline — "Electron ships an entire browser with the web app; smol machines ship an entire VM with the software" — tells you this is a better distribution mechanism, not a new behavioral primitive. Virtual machines have existed for decades. Containers have existed for over a decade. The `.smolmachine` artifact format is a clever packaging convention, and sub-200ms cold starts versus QEMU's 45+ seconds is genuinely impressive engineering, but nobody is going to use "smolmachine" as a verb. Compare this to Venmo, which created peer-to-peer mobile payments as a behavior that didn't previously exist, or to the Creator Fund ISAs where we literally invented people-as-investable-assets. smol machines is "faster, more portable VMs" — it competes on execution within the existing virtualization category rather than birthing a new one. The Electron parallel is the strongest bull case: Electron did become a distribution primitive for desktop apps. But Electron succeeded because millions of web developers already knew HTML/CSS/JS and desperately needed to ship cross-platform. Who is desperately demanding to ship entire hypervisor-isolated VMs with their CLI tool? That's a niche within a niche.

BinBin He's profile — ex-AWS Firecracker, sustained Rust/systems open-source work including ERA at 303 GitHub stars and envmap at 125 stars, FIRST Robotics mentor — reads as a calm, technically grounded engineer. The "sane" half of my framework is satisfied. But building portable lightweight VMs that boot fast is... sensible. It's a reasonable person with a reasonable idea. My antenna goes up when a very rational person pitches something most smart people consider impossible — when I have to ask "what if they're right?" about a claim that sounds crazy. smol machines doesn't trigger that. Everybody already agrees fast VMs are useful. Fu Qiao's background is a black box — no education, no prior employment, no GitHub repos found — which makes it impossible to evaluate the co-founder relationship or assess the team's full capacity against a multi-platform hypervisor engineering surface that spans macOS Apple Silicon, macOS Intel, Linux x86_64, and Linux aarch64. That's four platform targets for what the YC page lists as a team of one.

AWS already has Firecracker. Google has gVisor. Docker has $393M+ of investment and massive ecosystem lock-in. smol machines is pure software built on top of someone else's open-source library, with no physical-world operations, no protocol-level network effects, and no moat that survives the question "could an incumbent build this in two quarters?" The `.smolmachine` format creates potential switching costs only if adoption scales massively — and at 2,500 GitHub stars, smol machines is still far from that threshold. When I backed Metropolis at seed, the thesis was clear: AI amplifies the parking business but you still need physical parking infrastructure that no software company can replicate. smol machines has nothing analogous. It's a feature, not a fortress.

Here's the strongest bull case: AI coding agents — Claude Code, Cursor, OpenAI's Codex — genuinely need sandboxed local execution environments. GitHub Discussions on the smolvm repo already reference this use case. If smol machines became the default sandbox layer for AI-generated code execution, the `.smolmachine` format could evolve from a packaging convention into an industry standard — a protocol, not just a product. 2,500 GitHub stars in what appears to be the first weeks of public launch suggests real developer love, and BinBin He's Firecracker pedigree gives him credibility to capture this moment. That's a legitimate path to something bigger. But here's why it doesn't flip me: E2B is already purpose-built for AI agent sandboxing with $21M in funding and Fortune 100 penetration, and AWS can offer Firecracker-based sandboxing natively in their cloud where most AI agents actually run. The local desktop angle is a differentiator but also a constraint — it limits smol machines to a subset of developers who specifically need local isolation rather than cloud sandboxes. The condition that would flip me to Invest: signed integration partnerships with two or more major AI coding platforms (Anthropic, Cursor, OpenAI) adopting `.smolmachine` as a supported sandbox format. That would signal protocol-layer adoption rather than tool-layer usage, and it would create the kind of ecosystem lock-in that changes the structural durability picture entirely.

smol machines scores lowest on consensus mispricing and primitive creation — the two dimensions that matter most in my framework. Developer infrastructure is consensus-hot, and faster portable VMs are an execution improvement in an existing category, not a new behavioral primitive. BinBin He is a credible founder in the right domain, but the idea doesn't have the asymmetric risk/reward profile I look for — the kind where the market has priced something at zero because of discomfort or dismissal, and I get to ask "what if they're right?" Everyone already agrees this is useful. That's exactly the pricing problem.
