# InsForge -- Naval Ravikant Evaluation

Supabase has $500M in funding, $5B in valuation, 80K+ GitHub stars, and has already begun shipping MCP support for AI agents. InsForge's entire differentiation is a semantic layer — a context-optimized interface — over the same backend primitives Supabase already provides: Postgres, auth, storage, edge functions. When I look at this structure, I see the same architecture as the AI wrapper companies I consistently pass on. InsForge isn't calling someone else's API, which distinguishes it from the laziest wrapper pattern, but the core question remains: is a "better interface for agents" a company or a feature? Supabase can add agent optimization in a quarter. InsForge cannot replicate Supabase's ecosystem in a decade. The self-reported MCPMark benchmarks — 1.6x faster, 1.7x higher accuracy — come from InsForge's own benchmark suite with no third-party validation. Innovation without a moat is consumer surplus.

Tony Chang's Databricks networking infrastructure and Meta ads ML infrastructure internships are relevant to building backend systems but not specific to the problem InsForge is solving. He graduated UIUC in 2024 — roughly two years of professional experience. Hang Huang brings a Yale MBA and product roles at Amazon and Riot Games. Neither founder has years of obsessive engagement with how AI agents consume backend infrastructure, because this problem space barely existed before Anthropic shipped MCP in late 2024. When I invested in Perplexity, Aravind Srinivas had spent years at Google Brain and OpenAI building the exact neural architectures that powered AI search. InsForge's founders identified an opportunity through trend analysis — AI agents can't do backends well — rather than productizing non-replicable knowledge they already possessed. If Hang and Tony disappeared, another competent infrastructure team could rebuild InsForge's semantic layer without losing the core insight. That's the definition of replaceable specific knowledge.

Six YC rejections before acceptance. That number stopped me. InsForge's founders applied, got rejected, kept building, applied again — five times — with $300/month in revenue at the fifth rejection. They relocated from Seattle to San Francisco for the batch. This is the strongest signal in the entire dossier. Agents optimize for optionality; principals burn ships. A founder who applies to YC six times while generating $300/month in revenue is not playing a game — they're trapped by their own obsession, which is exactly where I want founders to be. The "building in public" strategy that doubled InsForge's database count from 2,300 to 4,000+ between November 2025 and January 2026 reinforces this: they were growing through personal evangelism, not paid acquisition. Sahil Lavingia kept building Gumroad after failing to raise a Series A. That same energy is visible here.

InsForge's leverage architecture is structurally sound but not structurally unique. BaaS is inherently code leverage — a 6-person team serving 4,000+ databases is a good ratio. Open source under Apache 2.0 creates distribution leverage. 7,700 GitHub stars, #1 Product of the Day on Product Hunt, #1 GitHub Trending during launch week — these are real adoption signals from a team spending zero on marketing. The problem is that every BaaS platform has this same leverage structure. InsForge's additional leverage claim — that the semantic layer makes agent interactions 30% more token-efficient — is a cost optimization, not a new form of leverage. Compare this to Notion, which created a genuinely new interaction paradigm (blocks) that became the product's identity. InsForge's semantic layer is an optimization of an existing paradigm (API calls), not a new one. Supabase adding MCP support is not a theoretical risk — it's an announced roadmap item from a company with 100x InsForge's resources.

The bull case is real and I want to be honest about it. InsForge has 4,000+ databases, 7,700 GitHub stars, and 2,000+ developers with a 6-person team — traction that many Series A companies would envy. The timing is genuinely right: AI coding agent adoption hit 51% daily usage among professional developers per MarketsandMarkets 2025 data, and the backend bottleneck is a real pain point. InsForge is first to build agent-native BaaS from the ground up rather than retrofitting it. First-mover advantage in developer tools can compound through community, integrations, and ecosystem lock-in — Supabase itself displaced Firebase partly through better developer experience, proving that interface philosophy can beat resources. InsForge's 3,546 commits and 32 releases from 6 people demonstrate the kind of shipping velocity I look for. The access democratization angle is genuine: InsForge enables non-backend developers to build full-stack applications through AI agents, expanding who can build software — a pattern that runs through AngelList, Gumroad, and Codecademy in my portfolio.

But the bull case requires believing that a semantic layer over commodity primitives is defensible against a $5B incumbent that's already moving into the space. I don't believe that. The compound interest test fails here: InsForge's advantages don't obviously compound over time in ways that widen the gap with Supabase. More databases create some switching costs, but Postgres is Postgres — migration tools exist. More agent interactions could improve the semantic layer, but InsForge hasn't demonstrated a data flywheel where usage improves the product in ways competitors can't replicate. The condition that would flip me to Invest: InsForge ships a proprietary data layer — something beyond standard Postgres — where agent interactions generate compounding intelligence that Supabase cannot replicate by adding MCP support. An agent-specific query optimizer that learns from millions of agent interactions, or a schema inference engine trained on InsForge's unique usage data. Until then, the semantic layer is a feature, and features get absorbed by platforms.

InsForge earns its score through genuine principal commitment (6 YC rejections, $300/month persistence) and competent leverage architecture (6-person team, 4,000+ databases), offset by thin specific knowledge (neither founder has non-replicable domain expertise) and existential platform risk (Supabase convergence). The decisive dimensions are Specific Knowledge Productization — where the founders' backgrounds are relevant but replaceable — and the gap between InsForge's interface innovation and a durable, compounding moat. Good founders, real traction, wrong structure for an outlier return.
