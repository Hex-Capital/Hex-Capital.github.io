# Clam (formerly Baseframe) -- Naval Ravikant Evaluation

The most striking fact about Clam is the competitive landscape it's entering. In 2025, four standalone AI security companies -- Lakera, Protect AI, CalypsoAI, Prompt Security -- were all acquired by large cybersecurity incumbents for $180M to $500M. This wave has a dual reading, and which reading you choose determines the entire investment thesis. Bullish: the field is cleared, incumbents are distracted by 12-18 months of integration work, and there's a vacuum for a focused, developer-first player. Bearish: the category is being absorbed into platform bundles, and "AI agent security" becomes a checkbox feature in every enterprise security suite within two years. I lean bearish. When Palo Alto Networks pays $500M for Protect AI, they're not going to let that capability atrophy -- they're going to ship it to every existing customer through their distribution machine. A two-person startup doesn't outrun that with a better proxy architecture.

On specific knowledge, the founders have relevant but shallow domain exposure. Anshul led AI evals and enterprise integrations at HappyRobot -- a production AI agent company serving logistics enterprises -- which means he encountered security gaps firsthand while deploying agents. That's genuine problem observation. Vaibhav built agent orchestration infrastructure at Augment Code and data pipelines at Sigma Computing. Both took a Computer Security course at Berkeley together. This is observational knowledge -- they saw where the pain was -- not the kind of deep, accumulated expertise that takes years to replicate. The real tell is the pivot: they launched BaseFrame (hardware design validation) in January 2026 and switched to Clam during the batch. When a team pivots into a hot category mid-batch, I ask whether they're following curiosity or following consensus. A founder with genuine specific knowledge in security would have started there.

The leverage architecture is the strongest dimension here and aligns with my portfolio preferences. A network-level firewall is infrastructure -- one team builds the interception layer, every AI agent in production flows through it. That's code leverage at the platform layer, exactly the picks-and-shovels position I favor. The credential injection mechanism -- keeping secrets entirely out of the AI's context by injecting them at the network level -- is an architecturally clean insight. If I squint, I can see the Stack Overflow pattern: a developer infrastructure layer that becomes more valuable with each agent interaction, building proprietary detection models from the traffic flowing through it. The problem is that the individual scanning components -- PII detection, prompt injection detection, malicious code scanning -- are individually well-understood and available in open-source tools like LLM Guard. The bundling at network level is an engineering choice, not a scientific breakthrough. A well-funded security team at Check Point could build a comparable proxy in a quarter.

The bull case I'd need to believe: AI agent deployment accelerates faster than incumbents can integrate their acquisitions, creating a sustained window where enterprises need standalone, best-of-breed agent security. Clam captures enough traffic during this window to build proprietary detection models that become genuinely hard to replicate -- the data flywheel kicks in before the incumbents catch up. The OpenClaw ecosystem grows explosively and Clam becomes the default security layer, the way Cloudflare became the default web application firewall. In that scenario, the network-level architecture compounds: every new attack pattern the firewall sees makes detection better for every customer. The founders' adjacent experience -- Anshul seeing the gaps at HappyRobot, Vaibhav building agent infrastructure at Augment Code -- proves sufficient to execute during the window. This is plausible. But it requires the incumbents to be slower than usual, the market to remain fragmented enough for a standalone player, and the founders to develop deep security expertise on the fly while competing against teams that have it natively.

This is AI security -- the hottest consensus category in enterprise software right now. Four companies acquired at nine-figure valuations in a single year. When every smart investor agrees that AI agents need security, the pricing already reflects the optimism. I'm not getting paid for being right first here; I'm arriving after the crowd has already bid up the thesis. The network-level approach versus model-level guardrails is an interesting architectural distinction, but it's not the kind of bet where informed people think you're wrong. Nobody is arguing against AI agent security. The contrarian bet in this space would be something like "AI agents don't need external security because the models themselves will become self-securing" -- and Clam is on the consensus side of that question.

The founders seem smart, technically capable, and their co-founder relationship -- Berkeley roommates, shared coursework, prior startup together -- is a positive signal for long-term collaboration. But the mid-batch pivot raises the question I always ask: is the passion for this specific problem, or for being founders? Hardware design validation and AI agent security are sufficiently different domains that the switch reads more like market-responsiveness than deep conviction. That's not disqualifying, but it's not the signal I look for when I'm betting my own money on a decade-long compounding game.
