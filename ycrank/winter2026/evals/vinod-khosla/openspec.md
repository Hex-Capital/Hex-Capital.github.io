# OpenSpec -- Vinod Khosla Evaluation

Here's what strikes me first about OpenSpec: the company has explicitly positioned itself as the modest version of an idea someone else is pursuing ambitiously. Tessl raised $125 million at a $750 million valuation to build "spec-as-source" where AI generates all code from specifications -- that's a bold thesis about the future of software development. OpenSpec, by contrast, describes itself as "lightweight," something that "supplements rather than replaces existing workflows." When I read a company positioning statement and the defining adjective is "lightweight," I know the founder is optimizing for adoption probability, not consequence magnitude. If OpenSpec achieves everything it's aiming for, the world gets... slightly more predictable output from coding agents. Developers save time on context management. That's useful. It might even be a real business. But nobody's life changes. No industry becomes structurally obsolete. The food system doesn't transform, the energy grid doesn't flip, the healthcare system doesn't reorganize. You've built a better workflow tool.

The competitive structure here reinforces the problem. GitHub launched Spec Kit with 71,000 stars and Microsoft's distribution to 100 million developers. Amazon built Kiro with 250,000 developers in its first months. BMAD Method has 39,000 stars as a permanently free community project. When incumbents are not only validating your category but actively building it -- and doing so with distribution channels orders of magnitude larger than yours -- that's not a signal of opportunity. That's a signal that you've identified a feature, not a company. My best investments were in companies that incumbents *mocked*. Cisco's CTO told me they would never build a TCP/IP router for public telecom networks. That dismissal was the signal that Juniper was attacking something real. When Microsoft, Amazon, and community projects are all rushing into your space, nobody's dismissing spec-driven development. It's consensus. And I don't invest in consensus.

Now let me address the bull case, because the timing dimension here is legitimately strong. AI coding agents are on an exponential trajectory, and the shift from writing code to specifying what to build is a real inflection point. Tabish identified this early -- starting after Claude 3.5 Sonnet's release, before most developers recognized that prompt alignment would become the bottleneck. 27,800 GitHub stars in seven months is evidence of genuine developer need. The Terraform analogy is tempting: HashiCorp built a multi-cloud neutral layer that succeeded despite every cloud provider offering competing infrastructure-as-code tools. If OpenSpec became the Terraform of spec-driven development -- the agent-agnostic standard -- you could imagine a meaningful enterprise business. For this to work, multi-agent neutrality would need to be a genuine enterprise requirement (debatable), and the spec format would need to accumulate enough ecosystem lock-in that switching costs compound (possible but YAML/Markdown specs have no inherent moat). If I believed the neutral-layer thesis, this could be a solid venture outcome. But "solid venture outcome" is exactly the territory I avoid -- either the money expires, or you build something that transforms an industry. A $500 million developer tools exit is not why I invest.

What I notice that a generic analyst would miss: Tabish has systematically de-risked this company in every dimension. Open-source, so no sales friction. IDE-agnostic, so no platform dependency. Lightweight, so no adoption barrier. Supplements rather than replaces, so no behavior change required. Each decision individually makes sense for adoption. Collectively, they've removed every source of friction *and* every source of structural advantage. The company has no proprietary technology, no switching costs beyond embedded spec files, and no mechanism to prevent GitHub from simply incorporating delta-spec functionality into Spec Kit. When you reduce every risk, you've also reduced the ceiling. The commercial layer (Fission AI) is pre-launch with no pricing, which means the monetization thesis is entirely hypothetical. Converting free open-source users to paying enterprise customers is one of the hardest problems in developer tools -- ask the dozens of open-source companies that built massive communities and modest businesses.

The founder shows real execution capability -- 34 releases, 50+ contributors, strong community engagement, 28 agent integrations built in seven months. That's not nothing. But execution speed on a developer tool is not the same as contrarian courage. Building something developers already want is engineering, not invention. The founders I back are attempting things the consensus considers crazy. Pat Brown was told plant-based meat would never taste like beef. Peter Beck was told New Zealand couldn't build orbital rockets. Tabish is building something that GitHub, Amazon, and multiple open-source communities all agree should exist. That agreement is precisely the problem.

I pass. The timing is right, the execution is competent, and the market is real. But the consequence of success -- even wild, beyond-expectations success -- is a useful developer productivity tool in a crowded and rapidly commoditizing category. Some problems are too important not to work on. Improving the prompt alignment of coding agents is not one of them.
