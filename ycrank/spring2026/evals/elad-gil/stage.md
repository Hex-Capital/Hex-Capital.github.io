# Stage -- Elad Gil Evaluation

The competitive landscape here tells a clear story, and it's not a favorable one for Stage. CodeRabbit has $88M in funding and $15M+ ARR building AI-automated code review. Greptile raised $29M and grew 10x in 2024 doing the same. Anthropic just launched a multi-agent code review tool for Claude Enterprise. Cursor acquired Graphite for reportedly over $290M in December 2025 -- meaning the dominant AI code editor now owns the leading code review workflow tool. And Haystack, a YC S24 company one batch earlier, is pursuing nearly the identical problem with a different visual metaphor. Stage is entering a market where the well-funded incumbents are converging on AI-automated review, the closest independent workflow competitor just got absorbed by the most important player in the space, and a direct overlap competitor has a head start. This isn't a non-obvious market at a structural inflection. It's a consensus-hot developer tools market with obvious sizing and intense competition at every tier.

The "why now" thesis -- that AI-generated code is creating a comprehension crisis for human reviewers -- is directionally correct. 84% of developers are using or planning to use AI coding tools, and the volume of unfamiliar code hitting pull requests is genuinely increasing. But here's what I think Stage's founders may be misreading: the structural response to AI-generated code isn't necessarily better human review tooling. It's AI-automated review. When the volume of machine-written code overwhelms human cognitive capacity, the market's natural equilibrium is machine-assisted verification, not reorganized diffs for humans to parse more efficiently. Stage is building a better filing cabinet at the moment when search is about to make filing obsolete. The adoption curve I track in code review points toward AI handling the first pass -- catching bugs, verifying patterns, flagging regressions -- with human review shifting to higher-order architectural and intent questions that don't require line-by-line diff comprehension at all. Stage's "structured chapters" solve a problem that may be shrinking, not growing.

The strongest bull case would require two things to be true: first, that human code review remains the critical bottleneck in software development even as AI review tools mature -- that regulatory, trust, or liability concerns keep humans in the loop for every meaningful change. Second, that "structured chapters" becomes a workflow standard the way stacked PRs became habitual for Graphite's users, creating switching costs that defend against GitHub eventually adding a similar native feature. If both are true, Stage could own the comprehension layer between AI code generation and human oversight, serving every engineering team that merges AI-assisted code. The Graphite acquisition actually helps this case -- it removed the closest independent competitor and left a gap in the market for review workflow innovation. A per-seat SaaS tool that becomes indispensable to every reviewer on a team has strong expansion dynamics. I can see the argument, but it requires the market to evolve differently than the current trend suggests.

The product-to-distribution trajectory is where Stage looks weakest through my framework. Code review is a feature, not a platform. GitHub, GitLab, and now Cursor+Graphite own the broader developer workflow -- the repository, the CI/CD pipeline, the editor, the deployment. Stage sits as a single-purpose tool adjacent to that workflow. Stripe started with payments but the developer integration became the distribution channel for Atlas, Billing, and Treasury. I don't see what Stage distributes through the review relationship beyond review itself. The product solves one problem -- diff comprehension -- without an obvious path to becoming the distribution channel for additional developer workflow products. That structurally caps the upside.

On the founders: Charles Pan and Dean Stratakos are solid Stanford CS engineers with Five Rings quant trading backgrounds, and Dean's AI specialization (Stanford MS in AI, built an in-house coding agent at Five Rings) is directly relevant. But "solid Stanford engineers who experienced a problem" describes half of YC's developer tools companies. There's no evidence of prior exits, no deep specialization in code review tooling beyond personal frustration, and no public traction signals -- no users, no revenue, no waitlist, no community. The cofounders clearly know each other well from overlapping at Five Rings, which reduces co-founder conflict risk, but nothing in the execution profile distinguishes them from the baseline YC applicant pool. I'd need to see weekly product velocity and early user love signals to move my conviction here.

The "structured chapters" concept is a UX innovation, not a technical moat. The dossier says it plainly: "Any well-resourced competitor could implement a similar organizational layer." This is what I'd call software-aware rather than software-driven -- you could strip out the technology and the core insight (organize diffs narratively) would survive as a design pattern anyone can copy. If the miracle is "build AI that auto-generates meaningful chapter structures from code diffs," that's achievable but not uniquely defensible. If the miracle is "get developers to switch from GitHub's native review UI to a standalone tool," that's historically one of the hardest adoption challenges in developer tools -- and it's the same miracle Haystack needs, with Haystack having a one-batch head start.
