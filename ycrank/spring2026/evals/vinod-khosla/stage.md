# Stage -- Vinod Khosla Evaluation

The most interesting thing about Stage is what it tells you about the current developer tools market — not about Stage itself. CodeRabbit has raised $88 million and is at $15 million ARR. Greptile raised $29 million. Graphite was acquired by Cursor for reportedly over $290 million. Haystack, one batch ahead at YC, is attacking the identical problem with a different visual metaphor. And now Stage enters with "structured chapters" for code review diffs. The competitive landscape is the signal here: when multiple well-funded companies and a direct YC competitor are already converging on the same user pain point, you're looking at a market that has been correctly identified but where the consequence of any single winner is... a successful developer tools SaaS company. I don't get out of bed for that.

Let me be direct about the consequence magnitude. If Stage succeeds completely — if every software engineer on earth reviews code through structured chapters — what changes? Engineers review PRs somewhat faster. They catch a few more bugs. Code quality improves incrementally. That's useful. That is also precisely the kind of outcome I filter out. Nobody's life fundamentally changes. No basic human need becomes radically cheaper. The world with a dominant Stage looks almost identical to the world without it — slightly more efficient, slightly less frustrated software teams. I have a term for this: a moderately profitable SaaS company. For most investors, that's a fine outcome. For me, it's a waste of a company's potential ambition.

The technical approach compounds the problem. "Structured chapters" is a UX innovation — it reorganizes existing data (git diffs) into a more comprehensible format. The dossier itself acknowledges: "The 'structured chapters' concept is a UX innovation, not a deep technical moat. Any well-resourced competitor could implement a similar organizational layer." I could not have said it better. This is a wrapper, not a wedge. Dean Stratakos has genuine AI chops — Stanford MS in AI, built a coding agent at Five Rings — and there's a version of this company where those skills produce something genuinely non-linear: an AI system that doesn't just organize diffs but actually understands code intent at a level that creates a step-function improvement in review quality. But that's not what the company describes. It describes organizing diffs into chapters. GitHub could ship that as a feature toggle next quarter.

The strongest bull case: AI code generation is creating an unprecedented comprehension crisis. Developers are reviewing code they didn't write and don't fully understand, at volumes that are exploding. The "why now" is real — 84% of developers use or plan to use AI tools, and the review bottleneck will only get worse. If Stage becomes the foundational interface for human-AI code collaboration, sitting at the chokepoint between AI generation and production deployment, it could matter enormously. Graphite's $290 million acquisition shows the market rewards workflow tools that become habitual. The Graphite exit also removed the strongest independent competitor, opening a window. For this bull case to hold, Stage would need to build something defensible enough that GitHub, Cursor, and Anthropic's own code review tools can't replicate — and the current "chapters" framing doesn't clear that bar. Dean's AI background could enable a pivot toward truly intelligent code comprehension, but I'm evaluating what they're building, not what they could hypothetically build.

The founders are solid — Stanford CS, Five Rings quant trading, professional overlap that suggests genuine trust. Dean's experience building an in-house coding agent at Five Rings is directly relevant and shows builder instinct. But these are consensus-elite credentials: Stanford plus top quant firm is the well-worn path, not the outsider energy I look for. I backed a 19-year-old dropout building weapons systems to compete with Lockheed Martin because the naivety was the advantage. Two Stanford quant traders building a better code review UI is competent, not audacious. I see no evidence of the rapid adaptation, first-principles surprises, or intellectual irreverence that signal a learning machine. They may have it — I just don't see it in the available data.

This is a clear pass for me. The consequence of success doesn't justify the attention cost, let alone the capital. If you're going to start a company and take the extraordinary risk that entails, build the thing that replaces the need for code review entirely — an AI system so reliable that human review becomes ceremonial — not the thing that makes the current process 40% more organized. The ambition level here is calibrated for a $200-300 million exit. That's someone else's investment thesis.
